01 ‚Äî Fundamentals

Core Concepts

üì¶

Repository

A repository (repo) is the container for your entire project ‚Äî every file, every change, and the full history of those changes. It lives in a hidden .git folder at your project root.

üì∏

Commit

A commit is a snapshot of your project at a specific point in time. Each commit has a unique hash, an author, a timestamp, and a message describing what changed.

üåø

Branch

A branch is a lightweight, movable pointer to a commit. Branches let you diverge from the main codebase to experiment or develop features without affecting the stable version.

üîÄ

Merge

Merging combines the histories of two branches into one. Git intelligently combines changes, and when two branches modify the same lines, it asks you to resolve the conflict manually.

üåç

Remote

A remote is a version of your repository hosted elsewhere ‚Äî on GitHub, GitLab, or another server. You push your changes to share them and pull to receive others' work.

üé≠

Staging Area

The staging area (index) is a preparation zone. You explicitly add files to stage them before committing ‚Äî giving you precise control over what goes into each snapshot.

üîÅ

Rebase

Rebasing replays your commits on top of another branch, creating a cleaner, linear history. Unlike merge, it rewrites commit history ‚Äî powerful but best used carefully.

üè∑Ô∏è

Tag

Tags mark specific commits as important ‚Äî typically release versions like v1.0.0. Unlike branches, tags don't move; they permanently point to one commit.

üîç

HEAD

HEAD is a special pointer that tells Git what you currently have checked out. It typically points to the tip of your current branch ‚Äî your "you are here" marker in history.

The Commands
You'll Use Daily

‚öôÔ∏è Setup & Init

$ git init
Initialize a new Git repository in the current directory

Clone a remote repository to your local machine

üìã Staging & Committing

$ git status
Show the state of the working directory and staging area
$ git add <file>
Stage a specific file for the next commit
$ git add .
Stage all modified and new files in the current directory
$ git commit -m "message"
Commit staged changes with a descriptive message
$ git commit --amend
Modify the most recent commit (message or content)
$ git diff
Show unstaged changes between working directory and index
$ git diff --staged
Show changes between staged files and the last commit

üåø Branching & Merging

$ git branch
List all local branches (current branch highlighted)
$ git branch <name>
Create a new branch at the current commit
$ git checkout <branch>
Switch to an existing branch
$ git checkout -b <branch>
Create a new branch and switch to it immediately
$ git switch -c <branch>
Modern equivalent of checkout -b: create and switch
$ git merge <branch>
Merge the specified branch into the current branch
$ git rebase <branch>
Reapply commits on top of another branch for a linear history
$ git branch -d <branch>
Delete a branch (safe ‚Äî won't delete unmerged branches)

üåç Remote Operations

$ git remote -v
List all remote connections with their URLs
$ git remote add origin <url>
Add a new remote named "origin"
$ git fetch
Download changes from remote without merging into local
$ git pull
Fetch and immediately merge remote changes into current branch
$ git push origin <branch>
Upload local branch commits to the remote repository
$ git push -u origin <branch>
Push and set upstream tracking for the branch

üîé History & Inspection

$ git log
Show the full commit history for the current branch
$ git log --oneline --graph
Compact, visual representation of branch and commit history
$ git show <hash>
Display details and diff for a specific commit
$ git blame <file>
Show who last modified each line of a file

‚Ü©Ô∏è Undoing & Recovery

$ git restore <file>
Discard changes in working directory (unstaged)
$ git restore --staged <file>
Unstage a file without discarding changes
$ git revert <hash>
Create a new commit that undoes a previous commit (safe)
$ git reset --soft HEAD~1
Undo last commit, keeping changes staged
$ git reset --hard HEAD~1
Undo last commit and discard all changes (destructive!)
$ git stash
Temporarily shelve uncommitted changes for later
$ git stash pop
Restore the most recently stashed changes

05 ‚Äî Best Practices

Pro Tips

‚ú¶ Commit Messages

Write commit messages in the imperative mood: Add login feature, not Added login feature. Keep the subject under 72 characters and explain why, not just what.

‚ú¶ .gitignore

Always create a .gitignore file to exclude build artifacts, node_modules, environment files (.env), and IDE settings. Use gitignore.io to generate one automatically.

‚ú¶ Atomic Commits

Each commit should represent a single logical change. Small, focused commits make code review easier, bugs easier to trace with git bisect, and history easier to read.

‚ö† Never Rebase Public Branches

Rebasing rewrites history. Never rebase commits that have been pushed to a shared remote branch. Use rebase freely on your local feature branches before merging.

‚ö† Never Force Push to Main

git push --force overwrites remote history and can cause teammates to lose work. If you must force push, use --force-with-lease as a safer alternative.

‚ú¶ Use Aliases

Speed up your workflow: git config --global alias.st status, alias.co checkout, alias.lg "log --oneline --graph --decorate". Small time savings add up.

‚ú¶ Pull Before Push

Always git pull --rebase before pushing to avoid unnecessary merge commits and ensure your changes apply cleanly on top of the latest remote state.

‚ú¶ Tag Your Releases

Use annotated tags for releases: git tag -a v1.0.0 -m "First stable release". Push tags explicitly with git push --tags. They serve as permanent milestones.

‚ö† Review Before Committing

Always run git diff --staged before committing to review exactly what you're including. It's easy to accidentally stage debug code, passwords, or unintended changes.
